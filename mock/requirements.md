# SETSUBI CONNECT — 機能要件・技術要件定義書

## 1. サービス概要

設備会社と物件管理会社をつなぐマッチングプラットフォーム。
一般的な一括見積もり型マッチングとは異なり、**運営者が審査・仲介**することで質の高いマッチングを実現する。
物件管理会社は「建物」単位で情報を登録し、将来の案件予測・定期メンテ営業に活用できる。

---

## 2. ユーザーロール

| ロール | 説明 |
|--------|------|
| 設備会社 | 空調・給排水・電気・消防等の設備工事を行う会社。審査通過後に利用可能 |
| 物件管理会社 | マンション・ビル等を管理する会社。設備会社への発注者側。審査通過後に利用可能 |
| 運営者 | プラットフォーム管理者。審査・マッチング・受注管理を行う |

## 3. 機能一覧

### 3.1 設備会社向け機能

| # | 機能名 | 説明 |
|---|--------|------|
| EQ-1 | 会員登録（審査申請） | 会社基本情報・事業情報・資格・実績等を入力し審査申請する |
| EQ-2 | ログイン | 審査通過後にメールアドレス＋パスワードでログイン |
| EQ-3 | ダッシュボード | 紹介案件数・商談中・受注確定・未読メッセージの統計、新着案件一覧 |
| EQ-4 | 紹介案件一覧 | 運営者から紹介された案件の一覧表示（ステータス別フィルタ） |
| EQ-5 | 案件詳細 | 案件情報（建物名・所在地・カテゴリ・希望時期等）の閲覧 |
| EQ-6 | 案件承認／辞退 | 紹介された案件を「承認する（商談開始）」または「辞退する」 |
| EQ-7 | チャット | 承認後に管理会社との1:1チャットルームが開設。テキスト送受信 |
| EQ-8 | メール通知 | 新規案件紹介時にメール通知を受信 |

**制約事項:**
- 管理会社一覧は閲覧不可（直接営業の防止）
- 案件は運営者からの紹介のみ表示される

## 4. 業務フロー

### 4.1 初期登録フロー

```
設備会社 → 会員登録フォーム入力 → 審査申請
                                      ↓
                              運営者が審査（承認 / 否認）
                                      ↓
                              承認 → ログイン可能に（メール通知）
```

```
物件管理会社 → 会員登録フォーム入力 → 審査申請
                                        ↓
                                運営者が審査（承認 / 否認）
                                        ↓
                                承認 → ログイン可能に（メール通知）
```

### 4.2 案件マッチングフロー

```
管理会社: 建物登録 → 案件登録（建物に紐づけ）
                          ↓
運営者: 案件一覧で「未マッチング」を確認
                          ↓
運営者: マッチング画面でカテゴリ×エリアから設備会社候補を抽出
                          ↓
運営者: 「紹介する」→ 設備会社に案件通知（メール＋システム内）
                          ↓
設備会社: 案件詳細を確認 → 「承認する」or「辞退する」
                          ↓
承認 → チャットルーム開設（設備会社 ↔ 管理会社の1:1）
                          ↓
管理会社: チャットでやりとり → 「この会社に発注する」ボタン
                          ↓
受注確定 → 以降の請求・施工管理はシステム外
```

### 4.3 再依頼フロー

```
管理会社: 過去の発注先一覧 → 「再依頼する」ボタン
                              ↓
再依頼フォーム（建物選択・案件名・内容入力）
                              ↓
設備会社に通知 → チャットルーム開設 → 発注確定
```

## 7. 技術要件

### 7.1 技術スタック

| レイヤー | 技術 | 備考 |
|----------|------|------|
| フロントエンド | Next.js (App Router) + TypeScript | React 19, RSC活用 |
| UIライブラリ | Tailwind CSS + shadcn/ui | ダークテーマベース |
| バックエンド | Next.js API Routes / Server Actions | フルスタック構成 |
| データベース | Supabase (PostgreSQL) | RLS（行レベルセキュリティ）活用 |
| 認証 | Supabase Auth | メール/パスワード認証 |
| リアルタイム通信 | Supabase Realtime | チャット機能 |
| ファイルストレージ | Supabase Storage | 会社資料PDF等 |
| ホスティング | Vercel | Next.jsに最適化 |
| メール送信 | Resend or SendGrid | トランザクションメール |

### 7.2 非機能要件

| 項目 | 要件 |
|------|------|
| レスポンシブ | PC優先（管理画面）。タブレット対応は任意 |
| ブラウザ対応 | Chrome / Safari / Edge 最新版 |
| セキュリティ | パスワードハッシュ化、RLS、CSRF対策、XSS対策 |
| パフォーマンス | 画面表示 3秒以内 |
| 可用性 | Vercel + Supabase のSLA準拠 |
| データバックアップ | Supabase自動バックアップ（日次） |

### 7.3 認証・認可

| ロール | 認証方式 | 認可ルール |
|--------|----------|-----------|
| 設備会社 | メール + パスワード | 審査通過（approved）のみログイン可。自社案件のみ閲覧 |
| 管理会社 | メール + パスワード | 審査通過（approved）のみログイン可。自社建物・案件のみ閲覧 |
| 運営者 | メール + パスワード | 全データ閲覧・操作可能。固定アカウント |

### 7.4 メール通知一覧

| トリガー | 送信先 | 内容 |
|----------|--------|------|
| 審査申請 | 運営者 | 新規審査申請があった旨 |
| 審査結果 | 申請者 | 承認 or 否認の結果 |
| 案件紹介 | 設備会社 | 新しい案件が紹介された旨 + 案件概要 |
| 設備会社承認 | 管理会社 | 設備会社が案件を承認しチャット開始した旨 |
| チャット受信 | 相手方 | 新着メッセージがある旨 |
| 発注確定 | 設備会社 + 運営者 | 受注が確定した旨 |
